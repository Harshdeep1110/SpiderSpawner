Executive SummaryGeneric wordlists like SecLists work well as a baseline but produce enormous noise on real targets. The vast majority of entries will never match anything on a specific application, resulting in slow scans, high false-positive rates, and missed context-specific endpoints.cwlgen (Context-Aware Wordlist Generator) solves this by scraping a live target and building a custom wordlist derived from what the application actually exposes: link paths, form parameter names, inline JavaScript variables, API route strings embedded in bundled JS, and CeWL-style keyword extraction. The resulting wordlist is smaller, higher-signal, and tuned to the target, making it far more effective for directory fuzzing, parameter discovery, and API endpoint enumeration with ffuf.2. Problem Statement2.1 Current approach and its limitationsThe standard workflow for web reconnaissance is to run ffuf with a generic SecLists wordlist (e.g. raft-medium-directories.txt, ~20,000 entries). This has three compounding problems:Signal-to-noise ratio is poor. A 20,000-entry generic list applied to a Rails app running on PostgreSQL will contain thousands of entries that could never match (e.g. PHP filenames, IIS-specific paths, legacy CGI routes).It misses target-specific vocabulary. A custom admin panel at /manage-reports or a parameter named redirect_to will not appear in any generic wordlist but may be discoverable by scraping the target itself.Modern JavaScript-heavy SPAs hide most of their API surface inside bundled JS. Standard crawlers and wordlists do not extract route strings from bundle files.Running large wordlists is slow and rate-limit-sensitive. Smaller, high-signal wordlists complete faster and are less likely to trigger WAF blocks.2.2 Target usersPenetration testers conducting authorized web application assessmentsBug bounty hunters doing recon on in-scope targetsCTF participants in web challenge categoriesSecurity students building hands-on tooling for their portfolio3. Goals and Non-Goals3.1 GoalsExtract all discoverable paths from a target page: href attributes, form action endpoints, script/img/link src paths, and CSS background-url references.Extract parameter vocabulary from form input names (including hidden inputs), query string keys, and URL path segments.Scan inline and external JavaScript files for quoted path strings, API base URL variables, JSON object keys, and variable/function names that suggest parameter names.Optionally integrate cewl-style keyword extraction (visible page text) for login brute-force or directory wordlist augmentation.Support session-based scraping for authenticated pages (cookie or Bearer token injection).Output a clean, deduplicated, sorted wordlist file and an optional JSON report listing extraction sources.Emit a ready-to-run ffuf command as final output.3.2 Non-Goalscwlgen is NOT a crawler. It scrapes a single provided URL. Recursive multi-page crawling is out of scope for v1.0.cwlgen does NOT perform any fuzzing itself. It only produces the wordlist.cwlgen does NOT execute JavaScript. SPA routes rendered only via client-side JS execution are not covered (that requires a headless browser, deferred to v2.0).cwlgen does NOT bypass WAFs or rate limits on the target during scraping.4. Feature Specification4.1 Input interface (CLI)cwlgen is a command-line tool. All options are specified via argparse flags.Usagepython cwlgen.py <target_url> [options]

  -o, --output FILE      Output wordlist path (default: wordlist.txt)
  -j, --json FILE        Also write JSON extraction report
  -c, --cookies STR      Cookie header string (for authenticated scraping)
  -H, --header KEY:VAL   Additional header (repeatable)
  -k, --keywords         Include cewl-style visible text keywords
  -t, --timeout INT      Request timeout in seconds (default: 10)
  --no-external-js       Skip fetching external JS files
  --min-len INT          Minimum word length to include (default: 2)
  --max-len INT          Maximum word length to include (default: 80)
  -v, --verbose          Print extraction breakdown by source type4.2 Extraction modulescwlgen runs five extraction modules in sequence. Each module outputs a set of strings. All sets are unioned and deduplicated at the end.ModuleWhat it extractsKey patterns / selectorsM1: Link extractionAll href, src, action attributesa[href], form[action], script[src], link[href], img[src]M2: Form analysisForm actions, input/textarea/select names, hidden input valuesform, input[name], input[type=hidden]M3: Inline JS scanQuoted path strings, API URL variables, JSON keys, var/let/const namesregex on <script> tag content (no src)M4: External JS scanSame patterns as M3, applied to fetched .js bundle filesAll <script src=...> resolved and fetchedM5: Keyword extractVisible text words (cewl-style, optional)soup.get_text() tokenized, filtered by length4.3 Regex patterns for JS scanningThe following patterns are applied to all JavaScript content in M3 and M4:Path strings: r'["\'](\/[\w\/\-_.?=&%]{2,})["\']'  — quoted strings starting with /API URL variables: r'(?:var|let|const)\s+\w*[Uu][Rr][Ll]\w*\s*=\s*["\']([^"\']+)["\']'JSON object keys: r'["\']([a-zA-Z_][a-zA-Z0-9_]{2,30})["\']:\s'Variable names: r'(?:var|let|const)\s+([a-zA-Z_$][\w$]{2,30})'4.4 URL normalizationAll extracted strings are normalized before being added to the output set:Absolute URLs on the same domain: strip scheme+host, keep path only.Absolute URLs on a different domain: discard.Relative URLs: resolve against base URL using urllib.parse.urljoin, then strip scheme+host.Path stripping: leading and trailing slashes removed. /admin/panel becomes admin/panel.Query strings: the path component is kept; query params are extracted separately as individual keys.Fragments (#): discarded entirely.javascript:, mailto:, data:: discarded.4.5 Output formatPrimary output: a plain text file, one word per line, sorted alphabetically, deduplicated. This is directly consumable as ffuf -w input.Optional JSON report (--json flag): structured extraction report for debugging and audit purposes.JSON report schema{
  "target": "https://target.com",
  "timestamp": "2026-06-06T10:30:00Z",
  "total_words": 94,
  "sources": {
    "hrefs": ["login", "about", "admin/panel"],
    "form_fields": ["username", "password", "redirect_to"],

}4.6 Final outputAfter writing the wordlist, cwlgen prints a ready-to-run ffuf command to stdout:[+] 94 words extracted → wordlist.txt[+] Run: ffuf -w wordlist.txt -u https://target.com/FUZZ -mc 200,301,302,403 -t 405. Technical Architecture5.1 Module structureFile layoutcwlgen/

Security and EthicsAuthorized use onlycwlgen is a security testing tool. It must only be used against targets you own, have explicit written permission to test, or that are designated as legal practice environments (HackTheBox, TryHackMe, DVWA, Juice Shop). Unauthorized use against live systems without permission is illegal under the Computer Fraud and Abuse Act (CFAA), the Computer Misuse Act (UK), and equivalent laws in India (IT Act 2000, Section 43/66). This tool is built for educational and authorized testing purposes only.6.1 Rate limitingcwlgen makes one HTTP request per page scrape plus one per external JS file found. It does not crawl recursively in v1.0. The fetcher enforces a configurable timeout (default 10s) and does not retry on failure. Users operating on slow or rate-limited targets should use --no-external-js to reduce request count.6.2 Credential handlingCookies and Bearer tokens passed via --cookies or --header are stored only in memory for the duration of the run and are not logged to any file by default. The JSON report does not include headers or credentials.6.3 Output handlingThe wordlist output file is written to the local filesystem only. No data is sent anywhere other than the target URL specified by the user.7.
